# 라온 프렌즈 AR 안내데스크 — 프로젝트 설계서

- **프로젝트명**: 라온 프렌즈 AR 안내데스크 (`raon-friends-ar`)
- **저장소**: https://github.com/wantaekchoi/raon-friends-ar
- **배포 URL**: https://wantaekchoi.github.io/raon-friends-ar/
- **목적**: 스마트폰으로 QR을 찍으면 앱 설치 없이 브라우저에서 라온 프렌즈(라오나·라오니·라옹)가 AR로 등장해 행사를 안내하고, 화면 안에서 설문에 답하면 응답이 구글 폼(스프레드시트)으로 자동 수집되는 웹 AR 서비스
- **제출 맥락**: 사내 AI 활용 경진대회 출품작
- **작성일**: 2026-07-16 (구 「라옹이 AR 안내데스크 설계서」 전면 대체)

> **구현 반영 노트 (2026-07-20 추가)**: 이 설계서는 2026-07-16 작성된 최초 설계 원안이며, 이후 구현 과정에서 실제와 달라진 부분이 있다. 원문 서술은 "당시 설계 의도"의 역사 기록으로 유지하고, 아래 각 장 상단에 실제와 달라진 지점만 짧게 표시했다. 확정된 현재 상태는 항상 [`CURRENT_STATE.md`](CURRENT_STATE.md)를 우선한다. 요약:
> - 3D 모델은 glb 변환 없이 fbx를 브라우저에서 직접 로드한다 (8장, ADR 0001 §3 참고).
> - AR 모드는 2종이 아니라 3종이다 — 오버레이·카드 마커·WebXR 바닥인식(Android 베타).
> - 설문은 6문항이 아니라 개인정보 동의 문항을 포함한 7문항이며 순서도 바뀌었다 (7.1 참고).
> - 화면 구성에 온보딩 1장·비AR 참여 경로·기념사진·언어 토글이 추가됐다 (6장 참고).
> - 개발 일정(11장)은 이미 전부 완료된 과거 계획이며, 현재는 배포 완료 상태다.

## 1. 전체 개요

한 문장으로: **QR 스캔 → 카메라 화면 위에 라온 프렌즈 등장 → 말풍선 안내 → 화면 내 설문 → 구글 폼 자동 전송 → 감사 인사**.

핵심 설계 원칙 네 가지.

1. **무설치**: 네이티브 앱 없이 모바일 브라우저(iOS Safari, Android Chrome)에서 동작. QR 코드 하나로 배포 완료.
2. **서버리스**: 백엔드 없음. 설문 응답은 구글 폼 `formResponse` 엔드포인트로 직접 전송되어 스프레드시트에 쌓임.
3. **이중 모드**: 카메라·권한 없이 무조건 동작하는 **쇼케이스 모드**(전 기기 보장) + 캐릭터 카드를 비추면 카드 위에 캐릭터가 등장하는 **카드 마커 모드**(임팩트 담당, 손 카드=MindAR NFT(Natural Feature Tracking, 자연 특징점 추적) · 바닥 대형 포스터=모서리 ArUco fiducial 자동 전환). 마커 인식이 불안정하면 쇼케이스로 폴백.
4. **재사용 킷**: 안내 멘트·설문 문항·폼 ID를 `src/config.js` 한 곳에서 관리 — 코드 지식 없이 문구만 바꿔 재배포하면 어떤 사내 행사·부스에도 재사용 가능. (심사 기준의 임팩트·확장성 어필 포인트)

**제출 스토리**: 2023년에 제작 후 잠들어 있던 3D 캐릭터 산출물을 AI(Claude Code)로 설계→구현→배포 전 과정을 진행해 웹 AR 서비스로 부활시킴. 신청서 문항 ①어디에 ②어떻게 ③무엇이 좋아졌나에 그대로 대응 (10장 참조).

## 3. 시스템 구성도

> **구현 반영**: `models/`는 glb가 아니라 fbx + jpg 텍스처를 그대로 배포한다(FBXLoader로 브라우저 직접 로드, ADR 0001 §3). 아래 다이어그램에는 없지만 실제로는 WebXR 바닥인식(hit-test)이 세 번째 AR 모드로 추가돼 있다.

```
[방문객 스마트폰 브라우저]
   │  QR 스캔으로 접속 (HTTPS 필수 — 카메라 권한 때문)
   ▼
[GitHub Pages]  ← main push 시 GitHub Actions가 Vite 빌드 후 자동 배포
   ├─ Vite 번들 (three.js + MindAR + 앱 코드)
   ├─ models/   raona.fbx · raoni.fbx · raong.fbx (+ 텍스처 jpg)
   └─ targets/  cards.mind (캐릭터 카드 3장 마커 데이터)
   │
   │  설문 제출 시 POST (no-cors)
   ▼
[구글 폼 formResponse 엔드포인트]
   ▼
[구글 스프레드시트]  ← 응답 자동 누적, 운영진 실시간 확인
```

## 4. 사용자 흐름 (시나리오)

> **구현 반영**: 실제 흐름에는 최초 방문 시 온보딩 1장(카메라 허용→바닥/카드 비추기→캐릭터 등장 안내), 설문 첫 문항의 개인정보 수집·이용 동의, 연락처 문항의 경품 추첨 안내가 추가돼 있다. 카메라를 쓸 수 없으면 시작 화면의 "카메라 없이 참여하기"로 AR 없이 설문에 바로 참여하는 경로도 있다.

1. **접속**: QR 스캔 → 시작 화면. [바로 만나기](오버레이) 또는 [카드로 소환하기](마커) 선택.
2. **카메라 허용**: 버튼 탭 이후에만 권한 요청(iOS 정책 대응). 거부 시 그라데이션 배경으로 대체하고 흐름은 계속 진행.
3. **캐릭터 등장**:
   - 오버레이: 라옹이 통통 튀며 등장 → 환영 인사.
   - 마커: 카드를 비추면 해당 캐릭터가 카드 위에 등장. 인식이 계속 실패하면 오버레이 전환 버튼 안내.
4. **안내 단계**: 머리 위 말풍선(타이핑 효과)으로 멘트 진행, 화면 탭/[다음]으로 넘김. 오버레이 모드는 이 시점에 라오니로 배턴터치. (선택: Web Speech API 한국어 TTS)
5. **설문 단계**: 라오나로 전환(오버레이 모드), 캐릭터는 하단에 축소 유지. 설문 카드가 1문항씩 슬라이드로 표시, 상단 진행 바.
6. **제출**: 마지막 문항 완료 시 구글 폼으로 자동 전송. no-cors 특성상 응답 확인 불가 → 1회 자동 재시도 내장.
7. **완료**: 진행 캐릭터의 감사 모션(오버레이 모드는 라오나, 마커 모드는 소환한 캐릭터) + "참여해주셔서 감사해요!" + [처음으로] 리셋(부스 태블릿 운영 대비).

## 5. 기술 스택

> **구현 반영**: "3D 모델" 행의 glb 변환은 실행되지 않았다 — fbx를 FBXLoader로 직접 로드하는 방식이 최종 선택으로 굳어졌다(ADR 0001 §3, NEXT_STEP.md 낮은 우선순위). "음성 안내(TTS)"는 불쾌한 골짜기 우려로 최종 **제외**됐고, 대신 WebAudio 신스 기반 효과음 4종 + 🔊 음소거 토글로 대체됐다(docs/specs/2026-07-16-improvements-design.md B팩). AR 모드도 여기 표에 없는 WebXR 바닥인식이 3번째로 추가됐다.

| 구분 | 선택 기술 | 선정 이유 |
|---|---|---|
| 빌드·개발 | **Vite** + GitHub Actions 배포 | npm으로 three.js/MindAR 버전 고정, dev 서버 HTTPS로 실기기 카메라 테스트 용이 |
| AR 기본 모드 | **카메라 오버레이** (getUserMedia 배경 + WebGL 오버레이) | 마커 불필요, iOS/Android 전 기기 동작 보장 |
| AR 마커 모드 | **MindAR** (이미지 트래킹, 무료 오픈소스) | 캐릭터 카드 위에 캐릭터 소환 — 카드가 곧 캐릭터 선택. (v1.3.0 하이브리드: 소환 후에도 세션 유지 — 카드 보이면 6DoF 부착, 놓치면 화면 유지·재스냅. iOS는 AR Quick Look usdz로 완전 고정 보완.) WebXR 평면인식은 iOS Safari 미지원, 8th Wall은 유료라서 제외 |
| 3D 렌더링 | **Three.js** (npm) | glb 로딩·애니메이션 표준, MindAR과 결합 검증됨 |
| 3D 모델 | **glTF 바이너리(.glb)** | 웹 표준 포맷. 보유 fbx를 Blender에서 변환 (8장) |
| 캐릭터 모션 | 코드 기반 모션 기본(등장 바운스·대기 숨쉬기·감사 점프) + 클립 있으면 AnimationMixer | 리깅만 있고 애니메이션 클립이 없어도 살아있는 느낌 보장 |
| 음성 안내 (선택) | Web Speech API (ko-KR) | 별도 파일·비용 없이 브라우저 내장 TTS |
| 설문 수집 | 구글 폼 **formResponse POST** (fetch, no-cors) | 서버 없이 스프레드시트 적재 (7장) |
| 호스팅 | **GitHub Pages** (Actions 자동 배포) | 무료 HTTPS, push만으로 배포 |
| 배포 | QR 코드 (배포 URL 인코딩) | 현장 접근성 극대화 |

## 6. 화면 설계 (4화면)

> **구현 반영**: 실제로는 아래 4화면 외에 온보딩 1장(최초 방문 시 카메라 허용→바닥/카드 비추기→캐릭터 등장 3스텝 안내), 비AR 참여 경로("카메라 없이 참여하기" → 설문 직행), 완료 화면의 기념사진 캡처 버튼, 시작 화면의 🌐 언어 토글이 추가돼 있다. S2의 "🔊 음성 토글"은 음성 안내가 아니라 **효과음 토글**로 구현됐다(5장 참고).

### S1. 시작 화면
- 타이틀 "라온 프렌즈 AR 안내데스크" + 3캐릭터 렌더 이미지
- [바로 만나기] (오버레이 모드) / [카드로 소환하기] (마커 모드)
- 하단 소형 안내: "카메라 권한이 필요해요"

### S2. AR 안내 화면
- 전체 배경: 후면 카메라 실시간 영상 (거부 시 그라데이션)
- 중앙: 3D 캐릭터 (등장 → 대기 모션, 오버레이 모드는 단계별 배턴터치)
- 상단: 말풍선 (안내 멘트, 타이핑 효과)
- 하단: [다음 ▶] / 진행 점 / 🔊 음성 토글 / (마커 모드) 오버레이 전환 버튼

### S3. 설문 화면
- 캐릭터: 화면 하단 구석으로 축소 (살아있는 모션 유지)
- 중앙: 설문 카드 1문항씩 (좌우 슬라이드 전환)
- 상단: 진행 바 (예: 3/6)
- 문항 유형: 단답 / 객관식 버튼 / 별점(1~5) / 장문

### S4. 완료 화면
- 진행 캐릭터의 감사 모션 (점프/하트 — 오버레이 모드는 라오나, 마커 모드는 소환한 캐릭터)
- "참여해주셔서 감사해요!" + 부가 안내 (운영 문구 교체 가능)
- [처음으로] 버튼 (다음 방문객 대비 리셋)

## 7. 설문 설계 및 구글 폼 연동

### 7.1 설문 문항 (v1 제안 — 확정 전 조정 가능)

> **구현 반영**: 아래는 최초 v1 제안이며, 실제 확정된 문항은 7개(개인정보 수집·이용 동의 문항 추가)이고 순서도 다르다 — **동의(필수) → 만족도 별점 → 인상 깊은 점(객관식) → 응원/의견(선택) → 성함 → 소속 → 연락처(선택, 경품 추첨용)**. 가벼운 선택형을 앞으로, 텍스트 입력을 뒤로 배치해 부스 회전율·중도이탈에 대비했다(`src/i18n.js`의 `survey.questions` 참고).

| # | 문항 | 유형 | 필수 |
|---|---|---|---|
| 1 | 성함을 알려주세요 | 단답 | ✅ |
| 2 | 소속(팀/부서)이 어디신가요? | 단답 | ✅ |
| 3 | 연락처(이메일 또는 전화번호) | 단답 | ⬜ (경품 추첨용) |
| 4 | 오늘 체험은 얼마나 만족스러우셨나요? | 별점 1~5 | ✅ |
| 5 | 가장 인상 깊었던 점은? | 객관식: AR 캐릭터 / 안내 콘텐츠 / 기술 완성도 / 아이디어 / 기타 | ✅ |
| 6 | 응원 한마디 또는 개선 의견 | 장문 | ⬜ |

### 7.2 구글 폼 자동 전송 원리

구글 폼은 문항마다 고유한 `entry.숫자` ID를 가진다. 폼의 제출 주소로 이 ID들을 담아 POST하면 정식 제출로 기록된다.

```
POST https://docs.google.com/forms/d/e/{FORM_ID}/formResponse
Content-Type: application/x-www-form-urlencoded

entry.1111111=홍길동&entry.2222222=플랫폼개발팀&entry.3333333=5 ...
```

- 브라우저 CORS 제한 때문에 `fetch(..., { mode: "no-cors" })`로 전송 — 응답 본문은 읽을 수 없지만 제출은 정상 기록됨.
- **entry ID 얻는 법**: 구글 폼 편집 → ⋮ → "미리 채워진 링크 받기" → 각 문항에 값 입력 후 링크 생성 → URL 속 `entry.xxxxx=값`이 문항별 ID.
- 앱에서는 설정 객체 하나로 관리:

```js
// src/config.js
export const GOOGLE_FORM = {
  formId: "1FAIpQL...",          // 폼 URL의 /e/ 뒤 부분
  entries: {
    name: "entry.1111111",
    org: "entry.2222222",
    contact: "entry.3333333",
    rating: "entry.4444444",
    highlight: "entry.5555555",
    feedback: "entry.6666666",
  },
};
```

### 7.3 구축 절차

1. **팀 공용 계정**으로 구글 폼 생성 — 7.1 문항 입력 (별점은 "직선 단계 1~5"). 공용 계정 소유로 두어 팀원 전원이 응답 스프레드시트에 접근 가능.
2. 응답 탭 → 스프레드시트 연결
3. 미리 채워진 링크로 entry ID 6개 추출
4. `src/config.js`에 formId + entry ID 기입
5. 테스트 제출 1건 → 스프레드시트 도착 확인

## 8. 3D 파이프라인

1. **변환**: `리깅적용/` fbx 3개 → Blender에서 **File > Export > glTF 2.0(.glb)** 변환. 내보내기 옵션: +Y Up, 애니메이션 포함, 텍스처 임베드.
   - **주의**: fbx에 텍스처가 연결돼 있지 않으면 `포즈/`의 텍스처 jpg를 Blender에서 수동 매핑 후 변환. 변환 직후 온라인 glTF 뷰어로 외형 검수.
2. **모션**: 애니메이션 클립이 없을 가능성이 높아(리깅만 적용) 코드 기반 모션을 기본으로 구현. 클립이 있으면 AnimationMixer 재생을 우선.
3. **용량**: glb당 **3MB 이하** 목표. 초과 시 `gltf-transform`으로 Draco 압축·텍스처 리사이즈.
4. **캐릭터 카드**: `랜더이미지/` 정면 컷으로 카드 3장 디자인(트래킹에 유리하도록 대비 높은 테두리·패턴 포함) → **MindAR 이미지 타깃 컴파일러**로 `cards.mind` 생성 → 인쇄용 PDF를 `docs/`에 포함.
5. 모델 로드 실패 시 임시 지오메트리 캐릭터로 대체 (개발·시연 안전장치).

## 13. 확장 아이디어 (발표 어필 포인트)

> **구현 반영**: 아래 중 "캐릭터 리액션 분기"와 "실시간 집계 대시보드"는 아이디어 단계를 지나 이미 구현·배포됐다(별점 4~5점 cheer/하트 파티클, 1~2점 sad → wave, `dashboard.html` 참고). "TTS 음성 안내"는 검토 후 불쾌한 골짜기 우려로 제외가 확정됐고, 대신 효과음 4종으로 대체됐다(5장 참고).

- 설문 응답에 따라 캐릭터 리액션 분기 (별점 5점 → 하트 뿅뿅)
- TTS 음성 안내 + 입 모션 연출
- 스프레드시트 응답 실시간 집계 → 부스 모니터 대시보드
- 카드 마커를 사내 제품 브로슈어·포스터로 확장 (제품 소개 AR)
- **단일 제품 홍보 킷**: OmniOne Badge 등 제품 출시·홍보 부스에 캐릭터·멘트만 교체해 투입
- **자사 주최 행사 투입**: 시큐업 세미나 등 대외 행사 안내데스크로 재사용 — 행사마다 쌓이는 참여 데이터 파이프라인
- **응답 데이터 AI 분석(보안 우선)**: 설문 추이는 외부 API가 아닌 로컬 업무용 PC의 sLLM으로 분석 — 데이터를 외부로 내보내지 않는 보안 회사다운 AI 활용
- 창립기념일 행사 등 사내 행사 현장 투입 — 실사용 실적 확보가 목표 (확정 전에는 사내 파일럿 피드백으로 대체 제시)
