# ADR 0001: 기술 선택

**날짜**: 2026-07-16
**상태**: 승인됨

라온 프렌즈 AR 안내데스크(design.md) 구현 과정에서 내린 핵심 기술 결정 4건을 기록한다.

---

## 1. 빌드 도구로 Vite 채택

**결정**: Vite 6을 빌드 도구로 사용한다. `npm install && npm run dev/build`로 개발·배포를 진행하고, GitHub Actions가 `npm ci → npm test → npm run build` 후 `dist/`를 GitHub Pages에 배포한다.

**이유**:
- three.js·MindAR 같은 npm 패키지의 버전을 `package.json`으로 고정할 수 있다.
- 개발 서버가 기본으로 모듈 번들링·HMR을 제공해 `src/` 하위 여러 모듈(config.js, flow.js, scenes/, ui/)로 코드를 나누기 쉽다.
- dev 서버를 HTTPS로 열면 실기기(스마트폰) 브라우저에서 카메라 권한 테스트가 바로 가능하다.

**대안 — 단일 HTML / no-build**: 빌드 과정 없이 `<script type="module">`로 CDN에서 three.js를 직접 로드하는 방식. 설정이 없어 더 단순하지만, 버전 고정이 어렵고 코드가 늘어날수록 여러 파일로 쪼개기 불편하며, GitHub Actions 배포 파이프라인(테스트 → 빌드 → 배포)을 구성할 지점이 없다는 단점이 있어 기각.

---

## 2. 이중 AR 모드 (오버레이 + 카드 마커)

**결정**: 마커 없이 항상 동작하는 **오버레이 모드**(`getUserMedia` 카메라 배경 + three.js 캐릭터 오버레이)를 기본으로 구현하고, 캐릭터 카드를 비추면 카드 위에 캐릭터가 등장하는 **카드 마커 모드**는 MindAR(이미지 트래킹, 무료 오픈소스)로 후속 구현한다.

**이유**:
- 오버레이 모드는 마커 인식 여부와 무관하게 모든 기기에서 보장되는 안전한 기본 체험이다.
- 카드 마커 모드는 "카드가 곧 캐릭터 선택"이라는 임팩트 있는 체험을 제공해 경진대회 심사에서 독창성 어필 포인트가 된다.
- 두 모드를 분리해두면 마커 트래킹이 불안정할 때 오버레이로 즉시 폴백할 수 있어 리스크가 낮다.

**대안**:
- **WebXR 평면 인식**: 표준 API라 매력적이지만 iOS Safari가 미지원이라 대회 현장(주로 스마트폰)에서 절반 가까운 기기를 놓친다는 치명적 제약으로 기각.
- **8th Wall**: 크로스플랫폼 마커 인식 품질이 우수하지만 유료 서비스라 경진대회용 무비용 프로젝트 취지에 맞지 않아 기각.

**후속 결정 (2차 개선 배치, E1)**: 한 번 기각했던 WebXR 평면 인식(hit-test)을 Android Chrome 베타 한정의 **세 번째 모드**로 다시 도입했다. iOS 미지원 제약은 여전히 유효하므로 오버레이 모드가 계속 기본 폴백을 담당하고, WebXR hit-test 지원이 확인된 기기(`navigator.xr.isSessionSupported('immersive-ar')`)에서만 "진짜 바닥에 소환" 버튼이 추가로 노출되는 방식이라 원래의 기각 사유(iOS 배제)와 충돌하지 않는다. 이로써 실제 AR 모드는 오버레이·카드 마커·WebXR 바닥인식 3종이 됐다(`src/scenes/webxr.js`).

## 3. FBXLoader 직접 로드 우선 (glb 변환은 후속)

**결정**: 보유한 리깅 fbx(`라옹리깅O.fbx` 등)를 Blender에서 glb로 변환하지 않고, three.js의 `FBXLoader`로 브라우저에서 직접 로드한다. 텍스처는 fbx에 연결이 없을 경우 이름 패턴 매칭으로 코드에서 수동 매핑한다. 로드 실패 시 임시 지오메트리 캐릭터로 자동 폴백한다.

**이유**:
- 로컬 개발 환경(macOS arm64)에 Blender·FBX2glTF·assimp 등 변환 도구가 설치되어 있지 않아, 지금 당장 glb 변환 파이프라인을 구축하는 것이 MVP 일정(오늘 배포)에 비해 과도한 선행 투자다.
- FBXLoader는 three.js 공식 addon으로 fbx를 별도 변환 없이 즉시 로드할 수 있어 리깅 자산을 그대로 재사용할 수 있다.
- 로드 실패에 대비한 폴백 캐릭터를 함께 구현해, 모델 파이프라인 문제가 전체 체험을 막지 않도록 안전장치를 뒀다.

**대안 — Blender glb 변환**: glb는 웹 표준 포맷이고 용량(Draco 압축)·로딩 속도 면에서 fbx보다 유리하다. 하지만 로컬에 Blender가 없어 지금 실행할 수 없으므로, MVP 이후 별도 계획(design.md 8장, 3D 파이프라인)에서 변환 도구를 설치·구축하는 후속 작업으로 미뤘다.

**후속 결정 (2026-07-20 기준)**: glb 변환은 결국 별도로 진행되지 않았다. FBXLoader 직접 로드가 "MVP 임시 방편"이 아니라 프로덕션의 최종 방식으로 굳어졌고, 배포된 서비스도 fbx + jpg 텍스처를 그대로 서빙한다(`public/models/*.fbx`). glb 변환은 여전히 미착수 상태로 [NEXT_STEP.md](../NEXT_STEP.md)의 낮은 우선순위 항목에 남아 있다.

## 4. 서버리스 구글 폼 연동

**결정**: 설문 응답 수집에 별도 백엔드를 두지 않고, 구글 폼의 `formResponse` 엔드포인트에 `fetch(..., { mode: 'no-cors' })`로 직접 POST해 스프레드시트에 적재한다. 폼 ID와 문항별 `entry.숫자` ID는 `src/config.js`에서 관리한다.

**이유**:
- 백엔드 서버·DB 없이 정적 사이트(GitHub Pages)만으로 설문 수집까지 완결되어 인프라 비용·운영 부담이 없다.
- 응답이 구글 스프레드시트에 실시간으로 쌓여 운영진이 별도 대시보드 없이 바로 확인할 수 있다.
- config 한 곳에서 폼 ID·entry ID를 교체하면 되므로 재사용 킷 설계 원칙과도 맞는다.

**대안 — 자체 백엔드**: 서버를 두면 제출 성공 여부를 확인할 수 있고(구글 폼 no-cors는 응답을 읽을 수 없음) 데이터 형식도 자유롭다. 하지만 서버 호스팅·인증·DB 스키마 관리 비용이 발생하고, 경진대회 일정(2주) 안에 정당화하기 어려운 과잉 설계라 기각. no-cors의 응답 미확인 리스크는 1회 자동 재시도로 완화한다(design.md 12장).
